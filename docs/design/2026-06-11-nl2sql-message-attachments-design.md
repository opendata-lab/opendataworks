# NL2SQL Message Attachments Design

## Current State

Conversation file infrastructure already exists (see
`2026-06-04-dataagent-conversation-files-design.md`):

- Each topic owns an isolated workspace; agent deliverables are written under
  `output/`, user inputs under `uploads/` (`core/topic_files.py`).
- The backend serves upload / list / download endpoints at
  `/api/v1/nl2sql/topics/{topic_id}/files[/{rel_path}]` with traversal,
  symlink, and HTML-XSS protection (`api/routes.py`).
- `NL2SqlChatV2.vue` has a collapsible right-side "会话文件" artifacts panel
  with grouped listing, preview, and download; it auto-refreshes when a run
  finishes.
- The system prompt (`prompts/data_agent_system_prompt.md`) already instructs
  the agent to write final reports under `output/`.

What is missing is the in-message experience:

- When a run generates a report (e.g. `output/sales_report.xlsx`), the answer
  text mentions the path but nothing in the message is clickable. The user must
  open the right-side panel and find the file manually.
- There is no association between an assistant message and the files its run
  produced. `da_agent_message` has `usage_json` / `error_json` but no
  attachments field, and nothing computes "files generated by this task".
- `renderMarkdown` (`chatMessage.js`) escapes and parses markdown but leaves
  relative file links (e.g. `[报告](output/report.html)`) pointing nowhere.

## Problem

Users want to download a generated report or file directly from the assistant
message that produced it, with one click, while the right-side panel keeps
showing all files of the conversation.

## Scope

In scope:

- Backend: per-run generated-file detection (workspace diff around task
  execution), persisted on the assistant message as attachments; attachments
  exposed in message payloads; a task-message lookup endpoint for the live
  flow.
- Frontend (`dataagent-frontend`): attachment download cards under assistant
  messages in both chat surfaces (`NL2SqlChatV2.vue` portal and
  `WidgetChat.vue` embed); markdown link rewriting so `output/...` /
  `uploads/...` links in answer text become working download links.
- Prompt: one line instructing the agent to reference generated deliverables
  as markdown links.

Out of scope:

- The right-side artifacts panel (unchanged; it remains the full-conversation
  file view).
- New file storage, ZIP/PDF aggregation, message→file DB foreign keys beyond
  the JSON snapshot, retroactive attachment backfill for historical messages.
- Adding an artifacts panel to `WidgetChat.vue`.

## Solution

### Generated-file detection (backend, primary path)

Attachment detection is a filesystem diff in `TaskCoordinator._run_task`,
independent of model output format and identical for local and sandbox-runner
execution (the workspace volume is shared):

1. Before `execute_task_stream`, snapshot visible workspace files
   (`rel_path -> (size, modified_at)`).
2. After the run returns (any terminal status), list again and keep files with
   `kind == "output"` (i.e. not under `uploads/`) that are new or changed.
3. Persist the resulting `WorkspaceFile` metadata list (capped, newest first)
   as `attachments_json` on the run's assistant message.

Helpers `snapshot_workspace_state()` / `diff_generated_files()` live in
`core/topic_files.py` next to the existing listing logic. Detection failures
are logged and never fail the run.

### Persistence and API contract

- New nullable `attachments_json` TEXT column on `da_agent_message`
  (Alembic migration).
- `update_assistant_message(..., attachments=None)`: writes the column only
  when a list is passed, so status-only updates never wipe attachments.
- Message payloads (`TopicMessage`) gain `attachments: List[WorkspaceFile]`,
  flowing through `GET /topics/{topic_id}/messages` for reload/history.
- New `GET /api/v1/nl2sql/tasks/{task_id}/message` returns the run's assistant
  message so the live stream path can pick up attachments right after a run
  ends without re-fetching the whole message page.

### Frontend rendering

- `hydrateMessageFromApi` carries `attachments` onto assistant messages.
- After the SDK event stream ends, `useNl2SqlChat.subscribe` fetches the task
  message (with a short retry, since the coordinator persists attachments just
  after the stream's terminal record) and sets `assistant.attachments`.
- Both chat surfaces render an attachment card row under the answer: file icon,
  name, size, and a direct `?download=1` link built from the existing
  `topicApi.fileUrl`.
- `renderMarkdown(text, { resolveFileHref })` post-processes rendered anchors:
  any workspace-relative href (no scheme, not root-absolute, not a fragment)
  is rewritten to a topic file download URL — `output/` is the prompt
  convention, not a requirement, and relative links have no other meaning in a
  chat answer. Absolute/external links are untouched. Each surface passes a
  resolver bound to its active topic.

### Prompt contract (secondary path)

`data_agent_system_prompt.md` report rules gain one instruction: reference the
generated file in the final answer as a markdown link, e.g.
`[销售报告](output/sales_report.xlsx)`. The filesystem diff remains the
reliable primary path; link rewriting only improves answers that follow the
convention.

## Interfaces

- `da_agent_message.attachments_json`: JSON array of
  `{name, rel_path, size, modified_at, content_type, kind}` (same shape as
  `WorkspaceFile`).
- `GET /api/v1/nl2sql/tasks/{task_id}/message` → `TopicMessage` (404 when the
  task or its assistant message does not exist or the topic is outside the
  request context).
- `TopicMessage.attachments`: optional list of `WorkspaceFile`.

## Tradeoffs

- Filesystem diff vs. tool-call tracing: the diff is execution-mode agnostic
  and needs no SDK hook changes, at the cost of attributing any concurrent
  workspace write to the running task. Topics run one task at a time, so this
  is acceptable.
- JSON snapshot vs. relational link table: attachments are a denormalized
  snapshot; if the file is later deleted from the workspace the download link
  404s. Acceptable — the right-side panel always reflects live state.
- Small race between the stream's terminal record and attachment persistence
  is handled with a bounded frontend retry instead of changing the stream
  protocol.

## Affected Stacks

- DataAgent backend: schema migration, store, coordinator, routes, prompt.
- DataAgent frontend: shared chat helpers, conversation engine, both chat
  surfaces.
- Main frontend / deploy: no changes (embed bundle and nginx proxying are
  unaffected).
