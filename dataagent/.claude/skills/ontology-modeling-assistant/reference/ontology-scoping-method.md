# 域划分与本体识别方法

本文档回答建模中最容易被跳过的问题:拿到一堆需求、文档和表之后,先建哪个域、哪些概念值得建成本体对象、哪些关系值得登记。按顺序执行四步:划分业务域 → 用典型问题收敛范围 → 识别本体对象 → 识别本体关系。每一步的产出直接作为 [`ontology-build-workflow.md`](ontology-build-workflow.md) 草案生成步骤的输入。

## 1. 划分业务域

先定域,再抽对象。域边界不清晰时,不要直接开始抽实体。

域边界判定来源,按优先级:

- 业务能力和业务流程:一个域对应一组目标一致的业务能力,例如风控审批、会员运营、供应链履约。不要按数据库、schema 或表名前缀切域。
- 术语口径一致性:同一个术语在域内只能有一种解释。如果"客户"在两个流程里口径不同,要么拆成两个域,要么在域内显式建成两个实体。
- 责任人边界:同一个域的口径应由同一批业务责任人拍板确认。
- 典型问题闭环:域内典型问题应能用域内对象闭环回答,不依赖另一个域裁决口径。

域分级,决定建模优先级:

- 核心域:直接承载业务差异和高频问数需求,优先建模。
- 支撑域:保障业务运转但不构成差异,按实际问数需求再建。
- 通用域:行业通用语义,例如组织、日历、地区。优先复用已有领域本体 skill,不重复建。

拆分与合并信号:

- 该拆:域内核心实体超过约 15 个;同一术语在域内出现多套口径;典型问题明显分属不同业务流程。
- 该合:两个候选域的典型问题大量互相引用对方实体;两个候选域的口径责任人完全相同。

一个领域本体 skill 只覆盖一个主业务域,对应元模型中的 `skill_covers_business_domain` 关系。跨域需求拆成多个 skill,不要做"全公司域"。

## 2. 用典型问题收敛范围

典型问题(competency questions)是范围裁决器:本体建成后必须能回答这些问题;反过来,支撑不了任何典型问题的候选对象和关系不建。

操作步骤:

- 和用户一起收集 5~15 个域内典型问题,覆盖四类意图:解释术语、找表找数、查指标、追踪关系。
- 把每个问题拆解为:名词(候选实体或属性)、动词和介词(候选关系)、统计表达(候选指标)、时间和过滤表达(候选口径)。
- 建模完成后逐条反向验收:每个典型问题能否用本体中的对象、关系和查询函数表达。不能表达的问题登记为 TODO,补对象或补关系后再验收。

## 3. 识别本体对象

候选来源,多来源交叉确认,单一来源不足以确认:

- 典型问题和用户陈述中反复出现的业务名词。
- 文档中有明确定义段落的概念、流程参与者、流程产物和业务事件。
- 表和字段注释中反复出现的业务名词;主键和外键指向的主数据概念。
- 血缘中被多个下游消费的核心表对应的业务概念。

实体判定清单,候选概念建成 `domain_entity` 前至少满足前两条:

- 有独立业务身份:业务上有唯一标识,或者可数("第 N 个订单"说得通)。
- 有独立生命周期或状态变化:会被创建、流转、关闭。
- 被至少一个典型问题当作主语或宾语引用。
- 在多份文档或多张表中稳定出现,不是一次性叫法。

实体 / 属性 / 指标 / 枚举值分流规则:

- 只依附于另一个对象、无独立标识 → `domain_attribute`,例如订单金额、客户状态。
- 是对实体集合的统计,有公式、聚合方式和时间粒度 → `domain_metric`,例如发布成功率。
- 是某个属性的固定取值 → 写进属性的取值说明,不单独建对象,例如订单状态的"已支付"。
- 一个概念既被当对象问又被当属性用时,按典型问题中的主要用法归类,把另一种用法写进 synonyms 或描述。

排除规则,以下不建为本体对象:

- 纯技术表:日志表、临时表、中间计算表、备份表,除非典型问题直接问它。
- 纯编码字典表:作为对应属性的取值来源说明,不建实体。
- 没有任何典型问题、文档定义或字段支撑的"未来可能用到"的概念。

数量控制:首版核心实体控制在 5~15 个。明显超出时回到第 1 步,检查是不是应该拆域。

## 4. 识别本体关系

候选来源:

- 典型问题中的动词和介词:"客户下了哪些订单"提示客户-订单关系。
- 文档中的流程顺序和归属表述:"审批通过后生成放款单"。
- 表之间的外键、惯用 join 键和血缘边。
- 用户对方向和基数的人工确认。

登记方式按关系性质分流:

- 实体之间的业务关系:默认直接登记 relation_kind=connects 的边,在 description 里写清业务性质、join 键和方向;只有当关系本身需要被解释、归类或复用时(对应字段字典中 kind=relation 的定义),才单独建 `semantic_relation` 对象。
- 实体拥有属性:relation_kind=has_attribute。
- 指标度量实体:relation_kind=measures。
- 术语或属性映射物理字段:relation_kind=semantic_mapping。
- 指标口径依赖字段、查询函数交接口径:relation_kind=caliber_rule。
- 文档和表对域、实体的支撑:relation_kind=supports 或 supports_scope。

可用的 relation_kind 枚举以 `scripts/ontology_schema.py` 的 `FIELD_DICTIONARY` 为准,用 `lookup_ontology.py` 查询确认,不要凭印象发明新枚举。

关系判定规则:

- 每条关系必须能支撑至少一个典型问题;纯粹"图好看"的关系不建。
- 字段同名、表名相似不构成关系。确认一条关系至少需要外键、血缘、文档表述或人工确认中的一项。
- 方向或基数不确定时保留 needs_confirmation,不要猜。

## 5. 产出与迭代

四步执行完后,交出以下清单,作为草案生成的直接输入:

```markdown
域划分结论:
- 业务域与分级(核心/支撑/通用):
- 域边界(包含/不包含):
- 本次建模的域:

典型问题清单:
- ...

候选对象清单:
- 实体(含判定依据):
- 属性:
- 指标:
- 排除项与排除原因:

候选关系清单:
- 关系(来源、方向、基数、确认状态):
```

每轮迭代新增对象或关系后,重新执行第 2 步的反向验收。
